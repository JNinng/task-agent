# Skill Loading Mechanism — Design Spec

Date: 2026-06-11

## Goal

Implement a two-layer skill loading mechanism matching the Python reference pattern from `learn-claude-code/s05_skill_loading.py`. Skills are stored as `SKILL.md` files with YAML frontmatter, listed in the system prompt (~100 tokens/skill) and loaded on demand via a `load_skill` tool (~2000+ tokens each).

## Architecture

```
internal/agent/
├── skill/
│   ├── skill.go           # Skill struct + SkillLoader
│   ├── skill_test.go      # Tests
│   └── load_skill_tool.go # load_skill tool (implements tools.Tool)
├── agent.go               # Modified: integrate SkillLoader, two-layer system prompt
├── runner.go              # No changes (Registry auto-includes new tool)
└── tools/
    └── tool.go            # No changes
```

### Skill File Locations

```
~/.task-agent/skills/        # Global skills
└── code-review/
    └── SKILL.md

<project>/.task-agent/skills/ # Project skills (override global)
└── git-workflow/
    └── SKILL.md
```

Project skills override global skills with the same name.

### SKILL.md Format

```markdown
---
name: code-review
description: Review code for correctness and quality
always_load: false
---

# Code Review Skill

Full skill instructions here...
Step 1: ...
Step 2: ...
```

## Components

### Skill struct (`skill/skill.go`)

```go
type Skill struct {
    Name        string // from frontmatter, falls back to directory name
    Description string // from frontmatter
    AlwaysLoad  bool   // if true, body injected into system prompt
    Body        string // markdown content after YAML separator
}
```

### SkillLoader (`skill/skill.go`)

- `NewLoader(globalDir, projectDir string) (*SkillLoader, error)` — scans both directories, merges (project wins)
- Recursively globs `*/SKILL.md` in each directory
- Parses YAML frontmatter with a simple `---` delimiter split (no external YAML library dependency)
- Falls back to directory name when frontmatter `name` is missing
- Returns error if a SKILL.md fails to parse (missing name, invalid YAML)

### load_skill Tool (`skill/load_skill_tool.go`)

Implements the existing `tools.Tool` interface:

| Field | Value |
|-------|-------|
| `Name()` | `load_skill` |
| `Description()` | `Load a skill's full instructions by name. Use this when a task matches an available skill.` |
| `InputSchema()` | `{ properties: { name: { type: "string" } }, required: ["name"] }` |
| `Execute()` | Returns `<skill name="X">\n<body>\n</skill>` or error string for unknown skills |

Registered into the existing `tools.Registry` — no dispatch changes needed.

## Two-Layer Injection

### Layer 1: System Prompt (always present)

```
You are a coding agent at <workdir>.
Use tools to solve tasks. Act, don't explain.

Skills available (use load_skill to get full instructions):
  - code-review: Review code for correctness
  - git: Git workflow helpers
```

Each skill costs ~100 tokens (name + description). Generated by `SkillLoader.Descriptions()`.

### Layer 2: Tool Result (on demand)

When the model calls `load_skill("code-review")`:

```
<skill name="code-review">
# Code Review Skill

Full instructions here...
</skill>
```

Content returned by `SkillLoader.Get(name)`. Unknown name returns `Error: Unknown skill 'X'.`.

### always_load Skills

Skills with `always_load: true` have their full body appended to the system prompt after the skill list:

```go
// In agent.go system prompt construction:
system := basePrompt + skillLoader.Descriptions()
for _, s := range skillLoader.AlwaysLoaded() {
    system += "\n<skill name=\"" + s.Name + "\">\n" + s.Body + "\n</skill>"
}
```

## Data Flow

```
Agent startup
  → SkillLoader.Load(globalDir, projectDir)
  → Merge: project overrides global
  → skillLoader.Descriptions() → appends to system prompt
  → always_load skills → appended to system prompt
  → load_skill tool → registered in Registry

Agent loop (unchanged)
  → Model sees skill list in system prompt
  → Model calls load_skill("X") when needed
  → Registry.Dispatch handles alongside other tools
  → tool_result returns full skill content
  → Model follows skill instructions
```

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Override strategy | Project overrides global | Per-project conventions take precedence |
| Load timing | Eager at startup | Skills are small; lazy loading adds complexity for no benefit |
| YAML parsing | Simple `---` split + manual parsing | Avoid external dependency; only 3 known fields |
| System prompt format | Name + description only (not body) | Keeps token count low for always_load=false skills |
| Error handling | `load_skill` returns error string | Matches existing tool error pattern; doesn't abort loop |
| Skill directory naming | Directory name = skill name (unless frontmatter overrides) | Convention over configuration |
